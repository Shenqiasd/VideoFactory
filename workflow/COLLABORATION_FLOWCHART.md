# Claude + Codex 协作流程图

本文档用可视化方式展示 video-factory 的五步协作流程。

---

## 🔄 完整流程图

```
┌─────────────────────────────────────────────────────────────────────┐
│                          开发者提出需求                              │
│                               ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │  Step 1: Requirements (需求澄清) - Claude 主导              │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ Claude:                                          │      │   │
│  │  │ 1. 提出澄清性问题（位置、格式、是否可配置）       │      │   │
│  │  │ 2. 分析影响范围和前置条件                        │      │   │
│  │  │ 3. 识别技术风险                                  │      │   │
│  │  │ 4. 定义验收标准                                  │      │   │
│  │  │ 5. 输出：workflow/steps/step1_requirements.md    │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  │                          ↓                                  │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ 开发者: 审查需求，确认无误                       │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  └────────────────────────────────────────────────────────────┘   │
│                               ↓                                     │
│                    ✅ 需求明确 + 开发者确认                          │
│                               ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │  Step 2: Design (方案设计) - Claude 主导                   │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ Claude:                                          │      │   │
│  │  │ 1. 探索代码库（Read/Glob/Grep）                  │      │   │
│  │  │ 2. 设计技术方案（提供2-3个选项）                 │      │   │
│  │  │ 3. 编写伪代码骨架                                │      │   │
│  │  │ 4. 评估影响范围（修改文件、测试、文档）          │      │   │
│  │  │ 5. 设计回退策略                                  │      │   │
│  │  │ 6. 输出：workflow/steps/step2_design.md          │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  │                          ↓                                  │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ 开发者: 审查方案，选择并批准                     │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  └────────────────────────────────────────────────────────────┘   │
│                               ↓                                     │
│                    ✅ 方案可执行 + 开发者批准                        │
│                               ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │  Step 3: Implementation (实现开发) - Codex 主导            │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ Codex:                                           │      │   │
│  │  │ 1. 读取设计文档 (step2_design.md)                │      │   │
│  │  │ 2. 按设计路径逐步实现                            │      │   │
│  │  │ 3. 每完成一个文件运行测试                        │      │   │
│  │  │ 4. 记录进度到 progress.md                        │      │   │
│  │  │ 5. 输出：代码 + step3_implementation.md          │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  │                          ↓                                  │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ Claude:                                          │      │   │
│  │  │ 1. 代码审查（风格、类型提示、错误处理）          │      │   │
│  │  │ 2. 检查测试覆盖                                  │      │   │
│  │  │ 3. 提出优化建议                                  │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  │                          ↓                                  │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ Codex: 根据审查建议优化代码                      │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  └────────────────────────────────────────────────────────────┘   │
│                               ↓                                     │
│                    ✅ 本地测试通过 + Claude审查通过                  │
│                               ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │  Step 4: Verification (验证测试) - Claude + Codex 联合     │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ Claude:                                          │      │   │
│  │  │ 1. 运行所有测试（pytest -q）                     │      │   │
│  │  │ 2. 服务健康检查（如有必要）                      │      │   │
│  │  │ 3. 对照验收标准逐条验证                          │      │   │
│  │  │ 4. 识别已知问题和阻塞项                          │      │   │
│  │  │ 5. 输出：step4_verification.md + 测试截图        │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  │                          ↓                                  │   │
│  │              ┌───────────┴───────────┐                      │   │
│  │              │   测试是否通过？       │                      │   │
│  │              └───────────┬───────────┘                      │   │
│  │                 失败 ↙         ↘ 通过                       │   │
│  │        ┌──────────────┐   ┌──────────────────┐             │   │
│  │        │ Codex:       │   │ 开发者:           │             │   │
│  │        │ 修复Bug      │   │ 审查验证报告      │             │   │
│  │        │ 重新提交     │   │ 决定是否发布      │             │   │
│  │        └──────┬───────┘   └──────────────────┘             │   │
│  │               │ 回到 Step 4                                 │   │
│  │               └──────────────┐                              │   │
│  └────────────────────────────────────────────────────────────┘   │
│                               ↓                                     │
│                    ✅ pytest全绿 + 验收标准满足                      │
│                               ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │  Step 5: Release (发布复盘) - Claude 主导                  │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ 开发者: 批准发布                                 │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  │                          ↓                                  │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │ Claude:                                          │      │   │
│  │  │ 1. 更新架构文档 (architecture.md)                │      │   │
│  │  │ 2. 追加执行日志 (progress.md)                    │      │   │
│  │  │ 3. 编写复盘总结                                  │      │   │
│  │  │    - 效能分析（实际vs计划耗时）                  │      │   │
│  │  │    - 经验总结（做得好的/可改进）                 │      │   │
│  │  │    - 后续行动                                    │      │   │
│  │  │    - 技术债务追踪                                │      │   │
│  │  │ 4. 提交代码（包含workflow文件）                  │      │   │
│  │  │ 5. 重置状态 (current_step.json → idle)          │      │   │
│  │  │ 6. 输出：step5_release.md                        │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  └────────────────────────────────────────────────────────────┘   │
│                               ↓                                     │
│                    ✅ 架构文档已更新 + 代码已合并                    │
│                               ↓                                     │
│                          ✅ 迭代完成                                │
└─────────────────────────────────────────────────────────────────────┘
```

## 🔁 异常处理流程

```
┌─────────────────────────────────────────────────────────────┐
│                     需求变更 (任意阶段)                      │
│                              ↓                              │
│  ┌────────────────────────────────────────────────┐        │
│  │ Claude: 识别需求变更                           │        │
│  │ → 回退到 Step 1                                │        │
│  │ → 标注 [需求变更] 在 progress.md               │        │
│  │ → 重新澄清需求                                 │        │
│  └────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                 设计缺陷 (在 Step 3-4 发现)                 │
│                              ↓                              │
│  ┌────────────────────────────────────────────────┐        │
│  │ Claude/Codex: 识别设计缺陷                     │        │
│  │                                                 │        │
│  │ 小调整 → 直接修改，记录偏离原因                │        │
│  │ 大调整 → 回退到 Step 2，重新设计               │        │
│  │                                                 │        │
│  │ 决策：Claude评估影响 → 开发者裁定              │        │
│  └────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                     测试失败 (Step 4)                        │
│                              ↓                              │
│  ┌────────────────────────────────────────────────┐        │
│  │ Claude: 分析失败原因                           │        │
│  │ ↓                                               │        │
│  │ Bug修复 → 分配 Codex 修复 → 重新验证          │        │
│  │ 设计缺陷 → 回退到 Step 2                       │        │
│  │                                                 │        │
│  │ 记录：step4_verification.md 追加失败原因       │        │
│  └────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                Claude/Codex 意见冲突 (任意阶段)              │
│                              ↓                              │
│  ┌────────────────────────────────────────────────┐        │
│  │ 1. Claude 说明建议理由                         │        │
│  │ 2. Codex 说明实现理由                          │        │
│  │ 3. 开发者做最终决策                            │        │
│  │ 4. 在 progress.md 记录决策过程                 │        │
│  └────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────┘
```

## 📊 角色职责矩阵

```

│ 角色   │ Step 1  │ Step 2  │ Step 3  │ Step 4  │ Step 5  │
├────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│ Claude │ ████████│ ████████│ ▓▓▓▓    │ ████████│ ████████│
│        │ 主导    │ 主导    │ 审查    │ 主导    │ 主导    │
│        │ 提问    │ 探索    │ 检查    │ 测试    │ 复盘    │
│        │ 分析    │ 设计    │ 建议    │ 验证    │ 归档    │
├────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│ Codex  │         │         │ ████████│ ▓▓▓▓    │         │

│        │         │         │ 实现    │         │         │
│        │         │         │ 测试    │         │         │
├────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│ 开发者 │ ▓▓▓▓    │ ▓▓▓▓    │         │ ▓▓▓▓    │ ▓▓▓▓    │
│        │ 确认    │ 批准    │ 关注    │ 验收    │ 归档    │

图例:
████████ = 主要职责 (80%+ 工作量)

```
